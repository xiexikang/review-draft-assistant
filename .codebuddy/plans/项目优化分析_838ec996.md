## 产品概述

对 Chrome 扩展 "AI 一键评价助手" 进行全面代码优化，涵盖代码重复消除、架构改进、错误处理增强、安全性加固、性能提升和可维护性改善。

## 核心优化方向

### 1. 消除严重代码重复（DRY 原则）

- `getProvider` 函数在 `background/index.ts` 和 `background/queue.ts` 中完全重复，且均重复导入全部 8 个 provider
- 8 个 Provider 实现中 `buildRequest` 有 80%+ 相同代码（url 拼接、openrouter 检测、header 构造、body 构造），仅 baseUrl、模型前缀、claude 特殊格式不同
- `recommendedModels` 在每个 provider 文件和 `SettingsTab.tsx` 中各维护一份，完全重复
- `setNativeValue` 函数在 `jd/fill.ts` 和 `taobao/fill.ts` 中完全重复
- `text()` 工具函数在 `jd/extract.ts` 和 `taobao/extract.ts` 中完全重复

### 2. 架构改进

- `OrdersTab` 组件 428 行，职责过重（订单管理、生成逻辑、填入逻辑、复制功能、UI 渲染），需拆分
- 短/中/长草稿渲染模板几乎完全相同，应提取为可复用组件
- 消息监听中 `message: any` 丢失类型安全
- Provider 注册机制低效：每新增 provider 需改动 3 处文件（`index.ts`、`queue.ts`、`SettingsTab.tsx`），应改为自动注册

### 3. 错误处理与健壮性

- `parseDrafts` 中 `JSON.parse` 无 try-catch，AI 返回非 JSON 时直接崩溃
- `fill` 函数中 Promise 无超时保护，content script 无响应时永久挂起
- `deepseek` provider 未处理 baseUrl 尾部斜杠（其他 provider 都处理了）
- `content/index.ts` 中 `sendMessage` 后未检查 `chrome.runtime.lastError`

### 4. 安全性

- API Key 通过 `chrome.runtime.sendMessage` 明文传递
- `manifest.json` 的 `host_permissions` 包含全部 AI 提供商域名，即使用户未使用

### 5. 性能

- 草稿生成纯串行（逐个 await），多订单场景应支持并发控制
- `draftsByOrderKey` 全量读写，数据量大时效率低
- `OrdersTab` 中内联函数/对象频繁创建，导致不必要重渲染

### 6. 可维护性

- `package.json` name 为临时名 "tmp-vite2"
- 淘宝新版页面选择器硬编码 CSS module hash（如 `.shopInfoOrderId--CVDgDEO2`），页面更新即失效
- `douyin`/`meituan` adapter 为空壳，无 TODO 标注
- 关键逻辑无单元测试

## 技术栈

- 现有：React 19 + TypeScript 6 + Vite 8 + TailwindCSS 3 + Chrome Extension MV3
- 保持现有技术栈不变，仅做代码层面优化

## 实施方案

### 策略一：提取共享基础设施（消除重复）

- 将 `getProvider` 函数和 provider 注册表抽到 `src/background/providers/registry.ts`，通过 `Record<ProviderId, ProviderAdapter>` 映射表实现，新增 provider 只需改一处
- 创建 `src/background/providers/base.ts`，提取 OpenAI 兼容格式的通用 `buildRequest` 逻辑（url 拼接、openrouter 检测、header 构造），各 provider 只需提供差异参数（baseUrl、模型前缀）
- 将 `recommendedModels` 的唯一数据源放在各 provider 文件中，`SettingsTab.tsx` 从 `registry.ts` 动态读取
- 将 `setNativeValue` 提取到 `src/platforms/shared/dom.ts`
- 将 `text()` 提取到 `src/platforms/shared/utils.ts`

### 策略二：组件拆分（架构改进）

- `OrdersTab` 拆分为：`OrderCard`（单订单卡片）、`DraftSection`（草稿展示+操作按钮）、`OrderToolbar`（批量操作栏）
- `DraftSection` 组件接收 draftLength（short/mid/long）和文本内容，统一渲染复制/填入/发表按钮

### 策略三：健壮性增强

- `parseDrafts` 包裹 try-catch，返回友好的 `Result<T, E>` 模式或抛出可读错误
- `fill` 函数添加 `AbortController` + 超时（如 10s）
- `deepseek` provider 补齐 baseUrl 尾部斜杠处理
- `content/index.ts` 的 `sendMessage` 添加 `lastError` 检查

### 策略四：性能优化

- 草稿生成引入并发控制（如 `p-limit` 或简易信号量），允许同时 2-3 个请求并发
- `draftsByOrderKey` 改为增量更新（只写变化的 key），避免全量覆盖
- `OrdersTab` 中的事件处理函数用 `useCallback` 包裹，减少子组件重渲染

## 实施注意事项

- Provider 重构需保持向后兼容：所有 provider 的 `buildRequest` 签名和返回值不变
- `claude.ts` 有特殊处理（原生走 `/v1/messages` + `x-api-key` header，openrouter 走 `/v1/chat/completions`），`base.ts` 需支持这种差异
- CSS module hash 选择器（如 `.shopInfoOrderId--CVDgDEO2`）无法根治，但可添加注释标注其脆弱性和备选策略
- 并发控制实现要简单，用 Promise pool 模式即可，避免引入外部依赖
